## 목적
거래명세서를 네가 입력·관리하는 도구. 본체 둘: **결제일 자동 계산** + **거래처·명세서·품목 자유 CRUD.**
거래처·명세서·품목을 언제든 생성/수정/삭제·저장, 거래처별 분류·합계·다가오는 결제 표시.

입력 = **수기 입력이 본체.** (엑셀 일괄 임포트는 코어 아님 — 아래 결정 참고.)

## 범위 경계
**포함** — 수기 입력 · 거래처/명세서/품목 CRUD(생성·수정·삭제·저장) · **카테고리 관리 목록 CRUD** · 결제일 계산·알림 · 거래처별 그룹핑·합계 · **테이블 뷰(정렬·필터·검색)**
**제외 (현 단계 — 안 만듦)**
- 인증·계정·멀티유저·서버 → 혼자 로컬
- 동시수정 락·감사 로그 → 사용자 1명이라 불필요
- 세금계산서·회계 전표 → 이카운트/더존 영역
- 멀티 회사(멀티테넌트) · 은행 연동 · 실결제
- **엑셀 일괄 임포트 → 코어 아님. 과거 데이터 이관용 선택 기능, 필요해지면 그때. (P0 아님)**

## ✅ 결정됨
- **배포 = Electron 포터블 .exe** (단일 윈도우 사용자, 더블클릭 원클릭, 설치 마법사 없음). better-sqlite3로 SQLite 직결.
  - 1명에게 만들어 전달. 서명 안 함 → 첫 실행 시 윈도우 "알 수 없는 게시자" 경고, "추가 정보 → 실행" 1회 (인계 메모).
  - 업데이트 = 새 .exe 전달 → 그 사람이 옛 .exe만 교체. 자동 업데이트 없음.
- **🗄️ 데이터 경로 = `app.getPath('userData')` (%APPDATA%) 고정. .exe 옆 저장 금지** — 이동/교체해도 데이터 보존. (P0)
- **🛡️ 데이터 불변식: 업데이트가 DB를 깨뜨리지 않게.** 스키마 버전 추적 · 마이그레이션 전 DB 백업 · 트랜잭션·전진형·가산적 · 금액 정수(원) 저장. 강제·상세는 coding-principles **P0 #6**. ⚠️ *규칙만 지금 박음 — 마이그레이션 엔진은 첫 스키마 변경 때 최소로, 미리 만들지 마라 (YAGNI).*
- 서버·인증·멀티유저 미구축 (YAGNI).
- **엑셀 = 수식 역설계 출처 (역할 끝남).** 실파일 3개에서 금액·세액·반올림 수식 다 추출 → `domain`에 고정. 세율 등 파라미터는 편집 데이터. 결제조건은 엑셀에 없음(앱이 새로 만드는 값).
  - **엑셀 임포터는 코어 아님 → 선택·나중** (과거 데이터 일괄 이관용). 만들 때도 이 회사 양식 1개만, 자동감지·멀티회사 금지.
- **구조 = 단일 세션.** 멀티에이전트 금지.
- **🔑 Repository 한 겹 (의무).** 데이터 접근은 `repository/`로만. UI·domain·parser는 SQLite **직접 호출 금지.**
  - 이유: 나중 멀티유저(서버+DB) 전환 시 교체 범위를 이 한 겹으로 가둔다.
  - ⚠️ 이 한 겹 외에 "나중을 위한" 서비스 레이어·인증 래퍼·제네릭 어댑터 추가 금지. 한 겹이지 프레임워크가 아니다.

## 데이터 모델 (다품목: 명세서 1장 = 품목 여러 줄)
```
Vendor          : { id, name, paymentTerms }
Category        : { id, name }                                                        // 관리 목록 (CRUD)
Transaction     : { id, vendorId, issueDate, dueDate, paymentStatus, memo }          // 명세서 헤더
TransactionItem : { id, transactionId, categoryId?, name, spec,
                    quantity?, unitPrice?, supplyAmount, taxType, vat, total }        // 품목 줄
```
**금액 규칙 (실엑셀 역설계 — 의존 사슬: 수량×단가 →(선택)→ 공급가액 → 부가세 → 합계):**
- `supplyAmount`(공급가액)**가 입력의 진실. `quantity`·`unitPrice`는 nullable 보조.**
  둘 다 숫자면 supplyAmount 기본값 = `quantity × unitPrice`(덮어쓰기 가능), 아니면 직접 입력.
  ⚠️ 수량·단가 필수로 만들지 마라 — 실데이터 절반이 총액만 있거나 수량이 "2박스" 같은 텍스트다.
- `vat`(부가세) = `taxType==='면세' ? 0 : round(supplyAmount × taxRate)` · `taxRate=0.10` (편집 가능 파라미터)
  반올림 = 0.5 올림(원 단위). 금액 양수라 `Math.round`와 동일.
- `total`(합계) = `supplyAmount + vat` · 명세서 합계 = Σ 품목 total

**카테고리 = 관리 목록 (거래처처럼 엔티티):**
- 추가·이름변경·삭제 모두 CRUD. 품목은 `categoryId`로 참조(미분류 허용 → nullable).
- 이름변경 = `Category.name` 한 곳만 고치면 참조 품목 전부 따라옴(FK라서).
- **사용 중 삭제 정책: 기본은 막기.** 그 카테고리를 쓰는 품목이 있으면 삭제 거부 + 건수 표시 → 재지정 후 삭제. (대안: 삭제 시 해당 품목 `categoryId=null`로 — 단 조용한 재분류라 기본 아님.)
- 초기 시드: 식자재/포장재/소모품/위생용품/기타. 실데이터엔 부재료·부자재·원재료도 있으니 **유저가 자유롭게 추가**. 임포터 만들면 미존재 이름은 거부 말고 자동 생성.

**고정 enum (값 추가는 코드 한 줄, 유저 UI 아님):**
- `taxType`: 과세/면세 (한국 부가세 2-값) · `paymentStatus`: 미지급/지급예정/지급완료
- 항목마다 선택·변경은 자유. "부분지급" 등 새 값 원하면 enum에 추가.

- 거래처·카테고리·명세서·품목 전부 CRUD — 모두 repository 경유 (P0 #5)

## 결제일 규칙 (= 거래처별 사용자 설정 데이터, 외부에서 받는 게 아님)
- `paymentTerms: { type: "net" | "dayOfMonth", value }`
  - `net` → 발행일 + value일 (net-30 등) · `dayOfMonth` → 매월 value일
- 네가 거래처마다 넣고 언제든 수정. 두 형태로 부족하면 그때 추가.
- **`dueDate`(결제일) = `paymentTerms`를 `issueDate`에 적용한 계산값 (명세서 단위, 1장당 1개).**
  엑셀엔 없는 새 값 = 앱의 존재 이유.
- **`paymentStatus`(미지급/지급예정/지급완료)는 별개 — 수동, "냈나" 상태.** dueDate(언제 내야)와 공존. 결제 단위가 명세서라 헤더에 둠.

## 조회 · 정렬 · 검색 (테이블 뷰 — "데이터베이스처럼")
품목 줄을 헤더(거래일자·거래처·결제상태)와 조인한 **평면 행 목록**이 메인 뷰. 모두 repository 쿼리(P0 #5, UI 생짜 SQL ❌).
- **정렬**: 거래일자·거래처·카테고리·공급가액·합계·결제상태·결제일 컬럼. ⚠️ 금액/수량은 **숫자 정렬**(정수 원 저장이라 자연 정렬됨), 날짜는 **날짜 정렬** — 문자열 정렬 버그 주의(테스트로 못박기).
- **필터**: 거래처 · 카테고리 · 결제상태 · 과세구분 · 월(yyyy-mm) · 결제일 기간.
- **검색**: 품목명·거래처명·비고 부분일치(LIKE). 거래처명은 trim 후 비교.
- 범위 가드: 단순 컬럼 정렬 + 위 필터 + LIKE까지. **쿼리 DSL·풀텍스트 엔진·동적 컬럼 빌더 만들지 마라** (P0 #2).
